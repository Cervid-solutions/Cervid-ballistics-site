# Cervid Ballistics — Project Brief

Use this to brief a new chat session. Attach `CervidBallistics.html` alongside it.

---

## What it is

A single-file HTML ballistics calculator for air rifles, rimfire, and centrefire. No server, no dependencies — open in any browser. Works on Safari on macOS and iOS.

## File

`CervidBallistics.html` — the complete, self-contained app (~920 lines).

Installed copies (after running `install.sh`):
- `~/Documents/Cervid Ballistics/CervidBallistics.html` (Spotlight-searchable)
- `~/Applications/Cervid Ballistics.app` (launcher, opens in Safari)

---

## Features

**Ballistics engine**
- RK4 integration in x-space (not time), so range-card rows hit exact yardage/metre marks
- G1 and G7 drag tables
- Spin drift (Miller formula), Coriolis effect, shooting angle (cosine rule)
- Wind deflection with clock-position input (3 = full crosswind from right, 9 = full left)

**Trajectory Truing**
- Fire at a known distance, enter the observed holdover
- Binary-search BC multiplier until the model matches reality
- Holdover input accepts mils, MOA, inches, or cm

**Environmental**
- Manual entry: altitude, temp (°C), humidity, pressure (inHg), wind speed, wind clock
- 🌍 Weather button: browser geolocation → open-meteo API → auto-fills all fields
- Works on Safari/iOS (no Bluetooth required)
- "Pressure from altitude" button auto-fills ISA pressure

**Unit system**
- Imperial / Metric toggle (all labels and values convert)
- Internal storage always in imperial; conversion on display

**Quick Presets**
- Built-in presets for air rifle (.177, .22, .25) and centrefire (.243, 6.5CM, .308); Rimfire mode ships empty (no built-in loads) — add your own via "+ Save"
- ✏ button on any preset (built-in or custom) loads it into the form for editing; saving hides the original and replaces it with the edited version
- Custom presets saved to localStorage
- Drag-and-drop reordering (all presets, built-in and custom, can be mixed)
- "Restore defaults" resets hidden/custom/order for the current mode

**Range card output**
- Columns: range, drop (in / cm), holdover (mils or MOA), velocity (fps / m/s), energy (ft·lbs / J), wind drift, spin drift, time of flight
- Optional "Spot Range" row — a single extra distance inserted anywhere in the table
- Interval auto or manual

## Architecture

Single HTML file: all CSS, JS, and markup inline. No build step.

Key JS sections (in order):
1. CSS variables / dark theme
2. Unit-system toggle (`setUnitSystem`, `syncLayout`)
3. Preset data (`PRESETS` object — air + cf arrays)
4. localStorage helpers (`loadCustom`, `saveCustom`, `loadHidden`, `saveHidden`, `loadOrder`, `saveOrder`)
5. Preset rendering (`getOrderedPresets`, `renderPresets`, `cancelPresetEdit`, `saveCurrentAsPreset`, `restoreDefaults`)
6. Preset loading (`loadPreset` — always stored in imperial, converts on load)
7. Ballistics engine (`dragCoeff`, `calculate`, `renderResults`)
8. Trajectory Truing (`applyTruing`, `resetTruing`)
9. Live weather (`fetchWeather` — geolocation + open-meteo)

State variables: `currentMode` ('air'|'cf'), `unitSystem` ('imperial'|'metric'), `twistDir` (1|-1), `editingCustomIdx`, `editingBuiltinIdx`, `lastResults`, `bcMultiplier`, `unitDrop` ('mils'|'moa').

---

## What has been deliberately removed

Kestrel 5700X BLE integration was built and then removed. It required Web Bluetooth (Chrome-only, not Safari/iOS) and wind speed encoding was never successfully decoded from the proprietary binary protocol. The live weather button is the replacement — it works everywhere.

## Known gaps / possible next steps

- No iOS home-screen icon / PWA manifest
- Presets are per-mode (air / centrefire); no cross-mode presets
- No export (PDF range card, CSV) — actually already covered by `exportCSV()`; a PDF range card is still missing
- Input `min`/`max`/`step` on `fPress` are static across both unit systems by design (already generous enough); everything else rescales via `BASE_BOUNDS`/`applyBounds()`
- Gyro stability (Sg) still has no velocity correction term. Miller's formula is referenced to 2800 fps; airgun loads run 700–950 fps. The new shape factor (see changelog below) only corrects for boat-tail/diabolo geometry, not the missing velocity term — this is the likely reason some slug loads still read "Over-stabilised" even after applying the boat-tail correction. Fixing this properly means adding a `(V/2800)^(1/3)` term, but that also needs validating against pellets (which already read plausibly without it) before trusting it for airgun velocities — flagged here rather than guessed at.

**2026-07-12 — Added GA, a diabolo-pellet-specific drag model, as a third Drag Model option alongside G1/G7 (all three files).** Followed on from an earlier wind-calculation investigation (see the wind-clock entries below): a centrefire preset checked against Applied Ballistics matched to within 0.01 MOA, but an air rifle preset showed a real ~7% gap against three independent airgun calculators, narrowing the cause down to G1 being a poor aerodynamic fit for diabolo pellets specifically rather than any bug in this app's math. Researched what the airgun ballistics community actually does about this: ChairGun's developers hit the same problem years ago and derived a named replacement drag table, "GA" ("airgun"), now also supported by Strelok Pro and widely treated as the de facto standard for diabolo pellets. The Cd-vs-Mach table now embedded in this app is the one shared by users extracting it from those apps' own data exports — a real, named, widely-used community model, but not an officially published spec sheet from Hawke/ChairGun, so it's documented in-code as the best available reference rather than an authoritative source.

Implementation reuses the existing table-driven architecture (`getCd(mach, tbl)` already interpolates any Mach→Cd table) rather than adding new machinery: `tbl` selection became a three-way ternary, and GA reuses the same primary BC field G1 already uses (both are "the BC number for whichever model is selected," unlike G7 which has its own optional field) — the only UI wrinkle was that the BC field's label was hardcoded to always say "(G1)," so it's now synced dynamically (`syncBcPrimaryLabel()`) to read "(GA)" when that model is active, called from every place the drag model can change (dropdown, preset load, mode switch, "+ New Load", and page-load preference restore).

Air mode's default now nudges toward GA instead of G1 for any load started from scratch, since that's the better fit for what most air rifle presets actually are; rimfire and centrefire still default to G1. Deliberately did *not* retag the eight existing built-in air rifle presets from G1 to GA, even though they're all diabolo pellets: their published BC figures were entered as G1-referenced numbers, and GA's Cd curve differs enough from G1's (lower through the subsonic band, sharply higher through transonic) that pairing an unchanged G1 BC with the GA curve would silently introduce a new, unverified error while chasing the original one — the same "don't curve-fit to a single data point" reasoning already applied during the wind investigation. GA is available for anyone who has a genuinely GA-referenced or pellet-specific BC (e.g. from ChairGun's own calculator, a manufacturer's GA-quoted figure, or future multi-point chronograph fitting), without disturbing the presets' existing, verified G1 pairing.

Verified via a Node/vm harness extracting the updated engine from all three files: syntax-checked; confirmed no duplicate ids; confirmed the dropdown carries exactly the three expected options in order; confirmed the BC label id and all six `syncBcPrimaryLabel()` call sites are present; and ran the JSB Exact .177 8.44gr preset through both G1 and GA at identical conditions, confirming GA produces sane, meaningfully different (not NaN, not wildly implausible) drop/drift/velocity figures consistent with the known shape of the GA curve — lower drag through the pellet's normal subsonic operating range, a much steeper transonic rise above it.

**2026-07-12 — Added a turret dial to Reticle View, showing a combined "dial + hold" solution across all three reticle patterns (all three files).** Requested addition: a way to enter how much has already been dialed on the scope's elevation and windage turrets, and see the *remaining* hold needed on the reticle for the rest — the way long-range shooters actually work when the full correction is more than they want to dial, or when they've dialed a round number and want to hold the remainder.

Two new inputs (Elevation, Windage) sit under the Reticle Pattern picker, each labelled with the sign convention that matches how turrets are actually marked — elevation positive = dialed up, windage positive = dialed right — rather than reusing the table's internal sign conventions (which are opposite-feeling for windage, as documented in the Reticle View entry below). Canonical storage is always in mils regardless of which pattern is selected, so a dial amount entered while viewing a mil pattern still displays correctly (converted via the standard 1 mrad = 3.43775 MOA factor) if the user then switches to the MOA Hash pattern — the physical dial amount doesn't silently change or reset when switching reticle illustrations.

The two axes needed different sign math, worked out from concrete examples rather than assumed symmetric: elevation's raw table value is already a *correction* ("positive = dial up"), so a matching turret dial subtracts directly from it. Windage's raw table value is a *drift effect* ("positive = drifts right"), and dialing right is the same kind of rightward push on point of aim as the wind itself — so it adds to the raw drift rather than subtracting, and the existing hold-position math (which already flips a positive combined value into a "hold left" instruction) handles the rest unchanged. Checked against worked examples before trusting it: dialing the exact amount needed reduces the remaining reticle hold to zero on both axes; over-dialing flips the remaining hold's direction, as it should.

**2026-07-05 — Lightweight second-point verification for Trajectory Truing.** Truing fits a single BC multiplier from one validated shot, which can mask a muzzle-velocity error that happens to look right at the truing distance but diverges further out. Rather than a full two-parameter (MV + BC) solve — which needs two accurate real-world shots and a coupled solver to be reliable — added a lightweight, advice-only check: a "Verify Second Point" sub-section inside Trajectory Truing with its own `fVerifyRange`/`fVerifyObs` inputs, separate from the truing inputs above them.

`verifyTruing()` does no fitting at all — it re-runs the existing `computeTrajectory()` at the new distance using whatever BC is currently in effect (trued or not, via `lastResults.params`), compares the predicted holdover against the shooter's observed value, and reports a qualitative verdict: "Holds up well" (≤5% delta), "Close, minor drift" (≤12%), or "Diverging — worth re-checking muzzle velocity" (>12%), plus the raw predicted/observed/delta numbers. If no truing has been applied yet, the result is still shown but flagged "(model is currently untrued)" so it reads as a sanity check on the raw model rather than implying a correction was made. `fVerifyRange`/`lblVerifyRange` convert between yd/m the same way `fTrueRange` does. `clearTruingState()` now also clears the verify inputs and status, so it resets correctly alongside the rest of truing state on preset switch, mode switch, and New Load.

Verified via the Node/vm harness: a self-consistent check (verifying against the model's own predicted value) reads as a perfect match; a 30%-inflated observed value correctly reads "Diverging"; verification works and is correctly labeled when run without any prior truing; empty/out-of-range inputs are handled without throwing; and the fields/status reset on preset switch along with the rest of the truing state.

**2026-07-04 — Projectile Shape selector for gyro stability.** Slug loads were reading "Over-stabilised" in the Gyro Stability card even when real-world experience says they shouldn't. Root cause: `millerSg()` infers stability purely from length/diameter/weight, which implicitly assumes a solid, roughly uniform-density projectile — it can't tell a boat-tail (sheds mass toward the rear, needs less spin than a flat-base of equal L/D) or a diabolo pellet (mostly hollow skirt, much less of its length is real destabilising mass) from a plain cylinder.

Added a "Projectile Shape" dropdown (Flat-base / Boat-tail / Diabolo) next to Drag Model. Selecting a shape applies an approximate, community-sourced Sg correction via `SHAPE_SG_FACTOR` (`flatbase:1.0, boattail:1.08, diabolo:1.20`), and the equivalent `sqrt(factor)` scaling is applied to the Min Twist recommendation for consistency. The applied correction is shown directly in the Gyro Stability card (e.g. "Over-stabilised · Boat-tail ×1.08") so it's never a silent adjustment. Shape is saved/loaded with presets (`shape` field, defaults to `flatbase` for any preset that predates this feature, so existing behavior is unchanged unless the user explicitly picks a shape).

Explicitly **not** implemented: a rigorous moment-of-inertia (McCoy-style) recalculation, which would need actual bullet/slug geometry (nose profile, boat-tail angle, hollow-point depth) rather than a flat multiplier. The chosen factors (1.08 / 1.20) are reasonable, commonly-cited approximations, not derived from first principles — flagged clearly in the UI caption and worth revisiting if it turns out to under- or over-correct for a specific slug.

Tested via the same Node/vm harness: confirmed the factor changes the reported Sg and status text, Min Twist scales by `sqrt(factor)`, and shape round-trips correctly through save/load of custom presets.
